## 1. パッド = 音楽のOS

> 「パッドで考えたり音楽を捉えたりするものとして考えれば、すべての楽器の考え方が統合できる」
> — うりなみさん（[[ハードコア・パッドスタイル 第2回 パッドのメリット・デメリット 他の楽器と渡り合うために|HPS第2回]]）

> 「パッドで音楽をやるときのOSを統合出来た」
> — 同上

パッドは楽器ではない。**音楽を捉えるためのOS**。

うりなみさんはギター、ベース、鍵盤、サックス、ドラムを演奏する。サックスは移調楽器（アルトはE♭、テナーはB♭——記譜のCと実音のCが違う）。鍵盤は白鍵と黒鍵で視覚情報が変わる。ギターは同一弦上で同時に複数の音が出せない。ドラムは和音が出ない。

パッドは違う。

- **同じ形 = 同じコード**。12キー全て、同じ押さえ方。鍵盤ではキーが変わるとフィンガリングが全て変わる
- **移調の壁がない**。サックス奏者がConcert PitchとWritten Pitchの変換に悩む必要がない。形を動かすだけ
- **Closed Voicingが弾ける**。ギターでは同一弦の制限でClosed Voicingに出来ないものがある。パッドにはその制限がない
- **Constant Structure**が5分で弾ける。Fmaj7→A♭maj7→D♭maj7→G♭maj7——鍵盤だと全部フィンガリングが変わる。1日掛かっても弾けないかもしれない。パッドなら形を覚えたら楽勝

だから、**ギターでもベースでもサックスでも鍵盤でも、パッドで考えればすべてが統合される**。各楽器に持ち替えた時に、パッドの思考をそれぞれの楽器に適用する。うりなみさんはそうして音楽のOSを統合した。

64 Pad Explorerは、このOSを可視化する装置。

## 2. なぜ4度のクロマチックか

> 「残念ながら4度のクロマチック以外はアドリブに必要な速度は出せないという結論です。フィンガリングの難易度が上がりすぎるからです」
> — うりなみさん（[[ハードコア・パッドスタイル 第2回 パッドのメリット・デメリット 他の楽器と渡り合うために|HPS第2回]]）

Launchpad初代（2009-10年頃）から2年間、複数のレイアウトを検証した。カスタムスクリプトでチューニングを変え、オープンチューニング的な可能性も探った。

結論: **4度のクロマチックだけがアドリブに必要な速度を出せる**。

| 4度のクロマチックの優位性 | 根拠 |
|--------------------------|------|
| **速度が出る** | フィンガリング難易度が低い。他のレイアウトはコード単体には良くても曲全体を弾くと破綻する |
| **ギター/ベースの知識が転用できる** | 4度チューニング = ギター/ベースの標準。タブ譜も実音がわかれば利用できる |
| **4度堆積和音が弾きやすい** | モーダルなものを弾く時に多用。テンションの追加も直感的 |
| **覚えることが少ない** | 覚えることを少なく済ませられる = 習得コスト削減 |
| **歴史的合理性** | 歴史的に残ってきたものは合理的。「負けた気がしましたね…」 |

**インキーモードはHPSでは扱わない**。キーをいちいち変えたらインプロができない。自由ではない。

## 3. 楽に弾く = 哲学

> 「とにかく楽が出来るように徹底的に考えるのがハードコアパッドスタイルの基本」
> — うりなみさん（[[ハードコア・パッドスタイル 第1回 フォームについて|HPS第1回]]）

> 「難しいことを練習するのではなくて、いかに簡単に弾けるようにするか考えることが大事です。どの楽器でも共通する考え方ですね」
> — 同上

「楽に弾く」は妥協ではない。**究極のプレイを目指すための設計原則**。

### パッド技術の本質（第1回より）

パッドの技術は極限すれば**2つだけ**：
1. **押さえる動作**
2. **ポジションを移動させる動作**

この2つを阻害する動作をなくしていく。それがフォームの全て。

### フォームの3原則

1. **肘から指先までで正三角形を作る** — 肩・肘・手首に負担を掛けない
2. **中指と親指を垂直にする** — 手の甲を水平にしない。叩く意識→手の甲が上→手首が立つ→速く弾けない
3. **手首を立てない** — 立てると可動域が狭くなり、衝撃が直接指に伝わる。回内・回外を使った連打もできない

### 両手の使い分けルール

- **横3パッド以内** → 片手。**4パッド以上** → 両手
- **縦1パッド以内** → 片手。**2パッド以上** → 両手
- **クロマチック上行 = 左手**（小指→人差し指方向は尺骨神経・虫様筋の構造上速い）
- **クロマチック下降 = 右手**

### 楽に弾けると何が起きるか

- リズムが良くなる（力みが取れると身体が自由になる）
- ダイナミクスが表現できる（弱く弾けることが必要）
- 練習効果が上がる（認知リソースが音楽そのものに向く）
- 怪我を防げる（RSI: 反復運動損傷を避ける）

パッドは複数のポジションで同じ音が弾ける = **一つの動作に固まらずに済む**。これはアコースティック楽器にないメリット。64PEがボイシングの代替配置を循環表示するのは、この哲学の実装。

### MPEフォームとの連続性

指の側面でパッドに触る → グリスやNote PBのコントロールがフォーム変更なしでできる。

> 「早い段階でマスターするとMPEを使ったプレイをしようとする時にフォームを変えずにすみます」
> — うりなみさん（HPS第1回）

うりなみさんはフォームの改造に2年かかった（Linnstrument導入後）。この苦労を他の人にさせないこと = **習得コスト削減の実践**。

## 4. 習得コスト削減 = 生涯のテーマ

> 「習得コストを減らすというのは私の生涯のテーマなので」
> — うりなみさん（[[出来ることを繰り返すことには意味がある|練習論]]）

うりなみさんは2度の事故で演奏能力の大半を失い、長いリハビリを経験した。箸の持ち方、歩き方、椅子の座り方、ギターの構え方——すべてを再学習した。**自分のOSを書き換えた**。

> 「自分のOSを変更したようなものですね」
> — うりなみさん（練習論）

だから「フォームの重要性」に神経質になる。だから「間違った動作の再学習コスト」を知っている。

練習論の核心:
- **出来ることを正確に、10分間繰り返す** — 出来ないBPMでやってはいけない。間違った動きを脳に覚えさせてしまう
- **1つの練習が複数の意味を持つようになる** — 自動化が進むと余裕が生まれ、ダイナミクス・リズムに注意が向く
- **疲れたら練習はするな** — 疲労時に間違った動作を強化するのが一番怖い。鉄則
- **いいイメージで終わる勇気** — ピーク・エンドの法則。成功体験で終わることが正しい神経回路の形成を促す
- **弾けない時でも上達できる** — シナプス結合は身体を動かさなくても促される。イメージトレーニングは本物

> 「学ぶことの素晴らしいことは奪われないことだって、B.B Kingが言ってるんですけど、本当にそうですよ。どんなに苦しい時でも主体性は保てる」
> — うりなみさん（練習論）

受験生の睡眠記録・勉強時間・正誤を365日記録してきた人間が語る練習論。音楽と受験は同根。**「他人の時間を浪費させないこと」が、うりなみさんのもう一つの使命**。

64PEの全ての機能は、この文脈の中にある。

## 5. 壁を壊す装置

> 「完成までの時間を圧縮するのではなくて、その人が出来るようになるまでの時間を圧縮する」
> — うりなみさん（[[ブログデータ/経験は圧縮されない。_絶望しないためにやれること|経験は圧縮されない]]）

64PEは「便利なツール」ではない。**音楽を学ぶ全ての壁を壊す装置**。

| 壁 | 壊し方 | 具体的な機能 |
|----|--------|-------------|
| **お金** | ブラウザ、全機能無料、PWA | Web版、音源内蔵 |
| **時間** | 携帯3ペインスワイプ、ショートカット | モバイル対応、1-7ダイアトニック |
| **機材** | テキスト入力（インクリメンタル）、中古MIDIキーボードでもOK | `/`キー入力、Web MIDI |
| **環境** | Lo/Hi Cut、エフェクト | 練習が楽しめなくなることを防ぐ |
| **操作** | セットアップゼロ、判断コストを掛けない | 音色保存、デバイス保存 |
| **言語** | 9言語対応 | パッドの情報は少ない。誰もやらないなら私がやる |
| **身体** | 色覚障害対応（Okabe-Ito） | 誰もが音楽にアクセスできるように |

| **楽器の壁** | ギター/ベース/ピアノからの入力 | 楽器ダイアグラム入力+コード判定 |
| **理論の壁** | 音が先、名前は後 | Plainモード→リアルタイムコード判定 |

**機能制限 = 主体性への壁 = NEVER**。学びに課金しない。MITライセンス — コードもオープン。コピーできる。しかし思想はコピーできない。

## 6. うりなみさんというミュージシャン

64PEを理解するには、作った人間を理解する必要がある。

### マルチ楽器奏者

| 楽器 | 位置 | 特記 |
|------|------|------|
| **ギター** | プロレベル。ファンク/ジャズギタリスト。Drop2、Shell voicing、テンションが語彙 | 長年の演奏経験。事故後も弾ける |
| **ベース** | 60-70年代ソウル・ファンクベースの音作り・奏法に精通 | オールドスクールファンクベースの記事あり |
| **鍵盤** | 後から覚えた。全キーすぐには弾けないが演奏はできる | 鍵盤なし生活を数年、パッドで代替 |
| **サックス** | マルチリード。**移調楽器** | 記譜と実音の変換問題を身体で知っている |
| **ドラム** | 子供の頃からの最初の楽器。**1度目の事故で諦めた** | アンラーニングに失敗した原体験 |
| **パッド** | Launchpad初代（2009-10年頃）から。Linnstrument, PUSH2/3, Move | 15年以上のパッド経験。日本で最もパッドの知識がある |

### なぜパッドに辿り着いたか

1. **ノン・キーボーディストに鍵盤を教えていた** → 管楽器奏者やギタリストが鍵盤でハーモニーを学ぶのは習得コストが高すぎる
2. **鍵盤以外でコードが弾ける楽器を探した** → パッドが最も手軽で、DAWとの親和性も高い
3. **全レイアウトを2年間検証した** → 4度のクロマチックが唯一のアドリブ対応解
4. **事故でギター/鍵盤の演奏能力を大きく失った** → パッドは身体に負担なく弾ける最後の砦
5. **複数の楽器を持ち替える中で「OS統合」に気づいた** → パッドで考えれば全楽器の理論が統合される

### うりなみさんの堀（Moat）

**理論 × 演奏 × 解剖学 × マルチ楽器 × 譜面 × プロ翻訳者**。この組み合わせは誰も持っていない。

64PEの機能は、このスキルセットから生まれている:
- **ボイシング計算の5×6リーチ制限** ← 演奏経験（人間の手で押さえられる配置）
- **Practical Sort（Lydian優先等）** ← 演奏経験（なっている音が正しい）
- **シェルボイシング→テンション拡張** ← 教授法（レッスンで積み上げたカリキュラム）
- **9言語対応** ← プロ翻訳者としての能力
- **Okabe-Ito配色** ← 解剖学の知識（人間の身体への配慮）

## 7. 複雑さはコード側、シンプルさはユーザー側

ユーザーが見るのはシンプルな画面。その裏で複雑な計算が動いている。

### 押さえたら音が出る

開いた瞬間から体験が始まる。音源内蔵、セットアップ不要。DAWは複雑すぎる — 音楽をやるのにDAWの使い方を学ぶ必要はない。

### コードを入れたら弾き方がわかる

`calcAllVoicingPositions` — 全組み合わせを網羅探索 → **5行×6列の手のリーチ制限**でフィルタ。数学的に正しい配置ではなく、**人間の手で押さえられる配置**だけを返す。

左右移動しても形が崩れない — ルートからの相対オフセットで「指の形のフィンガープリント」を取り、トランスポーズ時に同じ形を維持する。完全一致がなくてもフォールバック。**置き去りにしない。**

### 弾いたらコードがわかる（超高性能）

`detectChord` — 5段階スコアリング。完全一致 → omit5 → ベース+トライアド → ベース+テトラッド → テンション書き換え。6th+7th同時 = 13thに書き換える（音楽的文脈理解）。

**普通に作ろうとして、このロジックは作れない。** パラメーターの意味（5×6のリーチ、omit5の-5ペナルティ、スコアリングの重み）は全てうりなみさんの身体知から来ている。

### なっている音が正しい（Practical Sort）

| コード | Diatonic（教科書） | Practical（実践） |
|--------|-------------------|-------------------|
| I△7 | Ionian | **Lydian**（avoidゼロ） |
| iii7 | Phrygian | **Dorian**（暗すぎない） |
| vi7 | Aeolian | **Dorian** |
| viiø7 | Locrian | **Locrian ♮2** |

**ダンスミュージック、ブルース、ソウル、ファンクを辺縁・周縁としない。** Diatonicモードも消さない。**両方をトグルで並べた。** どちらかを排除しない = 個人を尊重する。

### 感覚に名前をつける、名前から感覚へ辿る

64PEの全ての入口は2方向を持つ。

**感覚 → 名前（Sound first, names second）**:
- Plainで適当に押す → 「これがCm7か」→ 調べられる
- MIDIで弾く → コード名がわかる
- ギター/ベース/鍵盤から入力 → パッドで見える

**名前 → 感覚**:
- Chordモードでコード入力 → 弾ける形で表示 → 音を聴く
- Available Scaleでスケール選択 → パッドの色でスケール音が見える
- ダイアトニックバーで1-7を叩く → 形と音で覚える

## 8. 開発の視野

### 生楽器奏者を取り込む

パッドの最大の強み = 楽器を問わず音楽を統合できること。

| ターゲット | 現在 | 目指す姿 |
|-----------|------|---------|
| **ギタリスト** | ギターダイアグラムからコード入力→パッド表示 | ギター判定→パッド表示（**Phase 1最優先**） + タブ譜自動生成 |
| **ベーシスト** | ベースダイアグラムから入力 | ベースライン→コード進行判定 |
| **鍵盤奏者** | ピアノ図から入力 | Web MIDI入力でリアルタイム判定 |
| **管楽器奏者** | — | Audio→MIDIコード検出（Web Audio API、マイク入力） |

**Audio→MIDIコード検出**: AnalyserNode + FFT → Chroma features（12音エネルギー分布） → ピッチクラス抽出 → 既存`detectChord()`で判定。マイク許可だけで動く。ギタリストが弾いたコードをパッド上に可視化できる。

### ギターエンジンの転用

pad-core共有ライブラリの`calcAllVoicingPositions`とジャンル別スコアリングは、Master Rhythm Chart（MRC）に転用できる。

**無料のコード譜にタブ譜が自動生成される** — これは前例がない。ギタリストがタブ譜付きコード譜を無料で使える世界。

### Web/Desktop境界

| Web（全機能無料） | Desktop（有料の対価） |
|------------------|---------------------|
| Audio→MIDIコード検出 | 市販プラグインの音で弾ける（VST3/AUホスティング） |
| コード譜、タブ譜、理論 | 超低レイテンシー |
| スケール・ボイシング全機能 | Push Universal Translator（MIDI Effect） |

**Webでできないものだけをデスクトップで提供する**。

### 開発ロードマップ（2026-03-08確定）

| Phase | 内容 | 状態 |
|-------|------|------|
| **1** | **ギター判定→パッド表示** — パッド人口を増やすネック。競合がいたら狙うポイント | Web版で最優先 |
| **2** | **有料版リリース** — Web版完成後。未完成で出すのはダメ | Web版完成後 |
| **3** | **ジャンル別拡張** — ファンク判定、ボッサコードベース等。HPS専用だが退会後も使える | 有料版後 |

## 10. 学習支援ソフトとしての核心

スケールは単体では意味を持たない。**キー設定があって初めて、コードの機能が決まり、使えるスケールが決まる。**

| 機能 | なにがわかるか |
|------|---------------|
| **Parent Scale逆引き** | このコードはどのキーに属しうるか = 転調の可能性 |
| **ピボットコード可視化** | 手動コード入力→ダイアトニックバーを消す→キーの呪縛から解放 |
| **スケールオーバーレイ** | このコードの上でどの音が使えるか、一目瞭然 |
| **テンションdimming** | 存在は示すが目立たせない。隠さない、でも導く（"dim but never hide"） |
| **オンコード + 転回形** | ベースラインの意識 → ボイスリーディング。ゴスペルのパッシングコード、鍵盤奏者以外が弱い領域 |

## 12. 安易に触らない

### 機能追加の判断基準

1. **壁があるか？** — ユーザーの障壁を壊す機能だけ追加する。「あったら便利」では作らない
2. **既存を壊さないか？** — キャッシュ3層、音声チェーン、座標系を理解してから触る
3. **バグ修正のWHYを読んだか？** — なぜそう直したかに設計思想がある

### 作らなかった機能

| 機能 | 理由 |
|------|------|
| **指番号自動生成** | 人間が教える価値。うりなみさんの身体知を自動化したら価値が消える |
| **機能制限による有料化** | 機能制限 = 主体性への壁 = NEVER |
| **Sequenceモード** | MRC（マスターリズム譜アプリ）で対応。一つのアプリに詰め込まない |
| **ユーザー認証** | ツール本体に認証は不要。自由 = 誰でもアクセスできる |
| **インキーモード** | キーを変えたらインプロできない。自由ではない（HPS第2回） |

## 13. ガイド改善の方針

### 思想は書かない。ツールに語らせる

機能・使い方・遊び方を示せば、思想は現れる。思想を言葉にしてツールに入れることは、**相手に読むコストを掛けさせる**こと。

### 書き方の原則

**「やってみたくなる文」を書く。理由は書かない。理由は使えばわかる。**

- NG: 「ダイアトニックコードを数字キーに対応させることで、音楽理論の習得コストを削減し…」
- OK: 「キーを選んで1〜7を押すだけ。好きな順番で叩いてみてください。気に入った並びはメモリーに保存できます」

### 初見ユーザーに伝えたい3つ（うりなみさん確認済み）

1. **押さえたら音が出る** — コードを入れたら弾ける形で表示される
2. **弾いたらコードがわかる** — 超高性能コード判定
3. **コードを入れたら弾き方がわかる** — 実際に押さえられる形、移動しても崩れない

### 構造

- **guide.html** = 機能・使い方・遊び方
- **ブログ記事** = WHYを語る場所（うりなみさんの言葉で。この文書が素材）
- **HPS** = 思想と身体知の本体
